# xiaogugit 接口调用说明

本文档描述 `server.py` 暴露的 HTTP 接口：**方法、路径、鉴权、参数与简要语义**。默认服务地址以 `http://127.0.0.1:8000` 为例，请按实际部署替换。

---

## 1. 通用约定

### 1.1 鉴权

除下列路径外，**其余接口均需登录**：

- `GET /login`、`POST /auth/login`、`POST /auth/logout`、`GET /health`
- 开发环境启用文档时：`/docs`、`/redoc`、`/openapi.json` 等

认证方式任选其一：

1. **请求头**：`Authorization: Bearer <access_token>`
2. **Cookie**：登录成功后服务端会写入 HttpOnly Cookie（名称见环境变量 `XG_AUTH_COOKIE_NAME`，默认 `xg_session`）
3. **API Key**：配置 `XG_SERVICE_API_KEY` 或 `XG_API_KEY` 后，使用 `X-API-Key: <key>`；同时兼容 `apikey: <key>` 与 `Authorization: ApiKey <key>`

获取 token：调用 `POST /auth/login`，响应体中的 `access_token` 即为 Bearer token。

### 1.2 请求格式

- 带 JSON 体的接口：`Content-Type: application/json`
- `project_id`：仅允许字母、数字、下划线、短横线
- `filename`：相对项目根路径，禁止 `..` 等路径穿越；禁止指向 `.git`

### 1.3 错误响应

业务或校验失败时，接口可能返回 **400**（如参数非法）、**404**（资源不存在）、**401**（未登录或登录失败）、**500**。部分错误在响应体中仅返回通用文案（如「请稍后重试」），详细原因以服务日志为准。

### 1.4 OpenAPI

开发环境默认开启时，可直接访问 **`/docs`** 交互式查看与试调所有接口。

## 2. 服务与认证

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/` | 服务信息与配置摘要（需登录） |
| `GET` | `/health` | 健康检查，返回 `status`、`env` |
| `GET` | `/login` | 登录页 HTML |
| `POST` | `/auth/login` | 登录，**Body**：`username`、`password` |
| `POST` | `/auth/logout` | 登出，清除 Cookie |
| `GET` | `/auth/me` | 当前是否已登录及用户名 |

**登录示例：**

```bash
curl -X POST "http://127.0.0.1:8000/auth/login" \
  -H "Content-Type: application/json" \
  -d "{\"username\":\"mogong\",\"password\":\"123456\"}"
```

## 4. 项目

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `POST` | `/projects/init` | **Body**：`project_id`（必填），`name`、`description`、`status`（可选） | 初始化或更新项目元数据 |
| `GET` | `/projects` | 无 | 项目列表 |
| `GET` | `/projects/{project_id}` | 路径 `project_id` | 项目详情（含文件列表等） |
| `POST` | `/projects/status` | **Body**：`project_id`、`status`、`operator`（可选） | 更新项目状态 |
| `GET` | `/projects/{project_id}/files` | 路径 `project_id` | 当前仓库内业务文件列表 |

## 5. 本体名称解析

| 方法 | 路径 | 查询参数 | 说明 |
|------|------|----------|------|
| `GET` | `/ontology-resolve` | `project_id`、`query` | 按名称/别名解析到具体 `filename`；启用 Redis 时优先走缓存索引 |

```bash
curl "http://127.0.0.1:8000/ontology-resolve?project_id=demo&query=发动机" \
  -H "Authorization: Bearer <token>"
```

## 6. 版本与时间线

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `GET` | `/versions/{project_id}/{filename}` | 路径含 **完整文件路径**；**Query**：`min_stars`（默认 0）、`sort_by`（`version` \| `stars`）、`order`（`asc` \| `desc`） | 单文件版本列表（过滤排序后） |
| `GET` | `/versions/{project_id}` | **Query**：`filename`（可选，指定则等价于单文件）、同上 `min_stars`、`sort_by`、`order` | 未指定 `filename` 时返回多文件结构 |
| `GET` | `/timelines/{project_id}` | 路径 `project_id` | 全部文件的简化时间线 |
| `GET` | `/version-detail/{project_id}/{version_id}` | 路径 `version_id`；**Query**：`filename`（可选，跨文件歧义时必填） | 版本详情（含提交与治理字段） |
| `GET` | `/version-read/{project_id}/{version_id}` | 同上 | 按版本读取快照 JSON；已删除版本返回 `deleted: true` |
| `GET` | `/projects/{project_id}/commits/{commit_id}` | 路径参数 | Git 提交详情 |

**注意**：路径中的 `{filename}` 可能包含子目录，需做 URL 编码（如 `a/b.json`）。

## 7. 写入、读取、删除、日志

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `POST` | `/write` | **Body**：`project_id`、`filename`、`data`（对象）、`message`、`agent_name`、`committer_name`、`basevision` | 写入新版本；首次写入 `basevision` 必须为 `0` |
| `POST` | `/write-and-infer` | 在 `WriteReq` 基础上可选：`inference_message`、`inference_agent_name`、`inference_committer_name` | 写入成功后调用概率服务，仅回填工作区 JSON 的 `probability`，不单独为新概率再升版本 |
| `POST` | `/delete` | **Body**：`project_id`、`filename`、`message`、`committer_name`、`agent_name`、`purge_history`（默认 `true`） | `purge_history=true` 时彻底清除该文件 Git 历史；`false` 时生成删除版本 |
| `GET` | `/read/{project_id}/{filename}` | **Query**：`commit_id`（可选） | 读最新工作区或指定提交快照 |
| `GET` | `/log/{project_id}` | **Query**：`filename`（可选） | 提交日志列表 |

---

## 8. 差异与回滚

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `GET` | `/diff` | **Query**：`project_id`、`filename`；**方式 A**：`base`、`target`（commit id）；**方式 B**：`base_version_id`、`target_version_id`（可同时不传 `base`/`target`） | 两提交间 diff，或两业务版本间结构化 diff |
| `GET` | `/version-diff` | `project_id`、`base_version_id`、`target_version_id`；**Query**：`filename`（可选） | 仅版本维度 diff |
| `POST` | `/rollback` | **Query**：`project_id`；`commit_id` 与 `version_id` **二选一** | 按提交回滚，或按版本 id 补偿式回滚 |
| `POST` | `/version-rollback` | **Query**：`project_id`、`version_id`；`filename`（可选） | 明确按业务版本补偿回滚 |

## 9. 推荐与点赞

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `POST` | `/version-star` | **Body**：`project_id`、`version_id`、`filename`（可选）、`increment`（默认 1） | 为某版本加星 |
| `POST` | `/version-unstar` | **Body**：`project_id`、`version_id`、`filename`（可选）、`decrement`（默认 1） | 减星 |

| `POST` | `/version-recommend/official/clear` | **Body**：`project_id`、`filename`、`operator`、`reason` | 清除官方推荐 |
| `GET` | `/version-recommend/official/history` | **Query**：`project_id`、`filename` | 官方推荐变更历史 |
| `GET` | `/version-recommend/community` | **Query**：`project_id`、`filename` | 社区推荐（星最高版本） |
| `GET` | `/version-recommend/community/history` | **Query**：`project_id`、`filename` | 社区版本排序列表 |
| `GET` | `/community-leaderboard` | **Query**：`project_id` | 项目内文件级社区排行榜 |

## 10. 典型 cURL（带 Bearer）

将 `<TOKEN>` 换为登录返回的 `access_token`：

```bash
# 写入
curl -X POST "http://127.0.0.1:8000/write" \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d "{\"project_id\":\"demo\",\"filename\":\"ontology.json\",\"data\":{\"name\":\"示例\"},\"message\":\"msg\",\"agent_name\":\"a1\",\"committer_name\":\"u1\",\"basevision\":0}"

# 读最新
curl "http://127.0.0.1:8000/read/demo/ontology.json" \
  -H "Authorization: Bearer <TOKEN>"

# API Key 调用
curl "http://127.0.0.1:8000/read/demo/ontology.json" \
  -H "X-API-Key: <API_KEY>"
```
